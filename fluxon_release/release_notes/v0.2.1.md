# Fluxon v0.2.1

`v0.2.1` is the first public Fluxon release. This release publishes the current integrated Fluxon foundation and the three public interface families built on top of it: key-value caching and remote procedure call (`KV/RPC`), message queue (`MQ`), and `S3`-compatible file and object caching (`FS`). It also includes the first public Quick Start image, packaged install artifacts, benchmark snapshots, and the current GitHub Pages documentation set.

## Release Summary

- First public Fluxon release.
- Unified Rust-based storage-and-transport foundation published as a reusable AI data plane.
- Public interface families published together: key-value caching and remote procedure call (`KV/RPC`), message queue (`MQ`), and `S3`-compatible file and object caching (`FS`).
- Quick Start image published for runnable `KV`, `MQ`, and `FS` demos.
- Packaged release artifacts published for installation and further evaluation.

## Foundation Capabilities In This Release

This first release publishes the current Fluxon base layer as the shared substrate behind all three public interfaces.

- End-to-end Rust hot path: connection handling, protocol encode/decode, state-machine progression, shared-memory management, and observability collection are pushed into Rust hot paths.
- Integrated storage and transport: storage and transport are optimized inside one unified data plane, with the cross-process shared-memory fast path prioritized where possible.
- High-performance inter-node transport: the current cluster data plane prefers `RDMA` inside the cluster, with automatic `TCP` fallback when needed.
- Dynamic NIC control from the UI: network interfaces can be enabled, disabled, and switched dynamically from the built-in GUI.
- Automatic inter-node relay: the system supports relay / forwarding across nodes and sub-clusters to reduce integration cost in more complex network topologies.
- Global memory allocation and governance: the release includes unified control over memory allocation, object lifecycle, capacity boundaries, and reclamation behavior.
- Unified role model: `master`, `owner_client`, and `external_client` are organized into a scalable tree topology that separates service logic from low-level data-plane governance.
- Unified object interface: multi-field objects are organized through one consistent system interface so higher-level APIs stay usable while lower-level optimization space remains open.
- Tensor-native zero-copy handoff path: the current release includes a zero-copy handoff path intended for frequent tensor-object reuse across cache and transport paths.
- Unified observability: `metric / trace / log` are consolidated through the `Prometheus` protocol and `Greptime`, with a built-in GUI for member state, topology, key metrics, and logs.
- Shared substrate across interfaces: key-value caching and remote procedure call (`KV/RPC`), message queue (`MQ`), and `S3`-compatible file and object caching (`FS`) reuse the same cache, transport, lease, capacity-governance, and observability foundation.

## Public Interface Coverage

### Fluxon KV/RPC

This release publishes Fluxon key-value caching and remote procedure call (`KV/RPC`) as the current public interface for world-model inference caches, state sharing, service-to-service calls, and tensor-object reuse.

- Local cache replicas with eventually consistent reads: the current read path prioritizes local fast-path hits while metadata synchronization continues asynchronously.
- Batched reclamation and hot-object management: invalid-object cleanup advances through `batch_delete`, and hot-object reuse is combined with `TinyLFU`.
- Coordinated `L2` and `L3` control for AI workloads: global objects stay indexed, discoverable, and reusable so duplicate residency across cache tiers can be reduced.
- KV and RPC on one substrate: state storage and service-to-service calls share one parameter organization, caching layer, and communication foundation.

### Fluxon MQ

This release publishes Fluxon message queue (`MQ`) as the current public interface for heterogeneous training, data-processing pipelines, and intermediate-state handoff across resource pools.

- `Lease`-based message retention: message retention is bound to the `channel`, so payloads keep bounded-time reliability before actual consumption.
- `channel`-level prefix statistics and capacity governance: the system maintains message-count and capacity-usage boundaries continuously for scaling and traffic control.
- Cross-cluster load-aware placement: payload placement can follow consumer-side location to shorten prefetch paths and stabilize throughput.
- KV-co-designed large-payload path: control-plane message shells and member metadata stay separate from large payloads, while large payloads stay on the `FluxonKV` data plane.

### Fluxon FS

This release publishes Fluxon `S3`-compatible file and object caching (`FS`) as the current public file and object cache for AI data and model files.

- Unified file / object / key-value cache substrate: `FS` directly reuses `FluxonKV/RPC` caching and communication capabilities, and maps files into `KeyValue` shards.
- `S3` forwarding access: the release includes object-storage access and forwarding for AI data and model files.
- Transparent Python file semantics: the current API path aims to preserve `open() / read() / write()` usage patterns while lowering system-call and cross-process overhead.
- Specialized optimization for small-file and large-file paths: the system differentiates concurrency and transport behavior by file granularity and read/write path.
- Large-scale cross-cluster migration path: the release includes the current unified path for large-scale data migration, together with caching, transport, and failure recovery.

## Quick Start Coverage

This release includes the first public `fluxon_quick_start:0.2.1` image. The Quick Start image exposes runnable demo flows for all three public interface families: key-value caching and remote procedure call (`KV/RPC`), message queue (`MQ`), and `S3`-compatible file and object caching (`FS`).

- `KV` Quick Start: includes interactive `put / get / del` operations and a KV Web UI.
- `MQ` Quick Start: includes interactive message production, a background consumer, and an MQ Web UI entry.
- `FS` Quick Start: includes shell-style file operations, an FS Web UI entry, and an `fs_s3` endpoint with default `admin / admin` Basic Auth for the demo environment.

## Benchmark Snapshot Included In This Release

This release publishes the current benchmark view described in the repository README and related test-stack materials.

- `RPC`: latency and throughput snapshots across different message sizes and concurrency levels.
- `KV`: current `TCP` benchmark snapshots show strong performance on `Read-affinity` and `Read-Zipf` relative to `MooncakeStore` and `Redis`; the current `put_only` bottleneck remains on inflight metadata deduplication rather than payload transport.
- `FS`: current benchmark snapshots show small-file reads and large-file writes already ahead of `Alluxio`, large-file reads roughly comparable, and small-file writes still with optimization headroom.
- `MQ`: this release mainly publishes the current scenario design and data-plane direction; automated runtime entrypoints live in `test_runner.py` and `fluxon_test_stack/`.

## Runtime Requirements And External Dependencies

- Linux only.
- Python `>= 3.10`.
- When building from source, the Rust toolchain follows `fluxon_rs/rust-toolchain.toml`, currently pinned to `1.93.0`.
- The minimum service plane requires `etcd` and `greptime`.
- `FluxonFS` features that persist task state for flows such as directory transfer and pre-scan also require `TiKV PD` and `TiKV`.
- Quick Start and runtime packaging workflows depend on Docker.

## Current Scope Notes

- This release publishes the current stable public contract of the repository, not a claim that every internal direction is already feature-complete.
- `KV/RPC`, `MQ`, and `FS` are all published together because they share one substrate, but their current benchmark depth and maturity are not identical.
- The current release content follows the capabilities and constraints described in the repository README and GitHub Pages docs at publish time.
