保持本文档简洁。
- 核心用户文档、开发文档和设计文档都在仓库内的 `fluxon_doc_cn/` 和 `fluxon_doc_en/` 下
- 详细的中英文文档写作规约索引见 `fluxon_doc_cn/dev_doc/开发者 - 3 - 文档写作规约.md` 和 `fluxon_doc_en/dev_doc/Developer - 3 - Documentation Writing Rules.md`
- `teststack` 有两个步骤：`start testbed` 和 `testrunner`
- `teststack` 支持 UI；`testrunner` 应负责 UI 的 authority 和 API surface，但 UI 应作为常驻服务运行，并复用下层的 ops 接口
- 本项目所有 Python 代码都必须兼容 Python `>= 3.10`
- 本项目中的 YAML 文件默认都是示例。不要直接修改它们；请为你的具体开发环境创建单独的 YAML 文件
- 长时间运行的命令请放到 `tmux` 里启动。不要直接在前台运行长生命周期服务
- Git 操作仅限基础的 `stage`、`unstage`、`commit` 和 `push`。不要使用其他 Git 操作
- 默认优先收束而不是兼容。除非任务明确要求，否则不要添加兼容层、废弃路径或别名
- 一个概念优先只保留一个正式名字。避免同义参数、重复入口和并行配置面
- 有意识地控制分支。优先采用类似 Rust enum 的小而显式、可穷举罗列的有限分支集合，而不是开放式扩散出一批近似重复分支
- 扩展一个 surface 时，优先把新情况折叠进已有的有限分支集合；如果确实必须新增分支，就让它保持显式、边界清楚、易于完整罗列
- testbed 作用域内的概念，命名里应显式带上 `testbed`。对仅属于 testbed 的 mode、port、root、workdir 等设置，避免使用过于泛化的名字
- 保持 `AGENTS.md` 和 `AGENTS_CN.md` 同步。修改仓库级 agent 规约时，除非任务明确说明，否则要及时同时更新中英文版本

## 文档站
- 文档站使用 Quartz。把 Quartz 视为位于 `.cached` 下的缓存构建工具，不要以 git submodule 方式引入
- 将仓库根目录的 `README.md` 发布为文档站首页
- 不要在 `fluxon_doc_cn/**` 或 `fluxon_doc_en/**` 下添加索引 `README.md`；应使用真实内容页和自动生成的导航
- GitHub Pages 产物必须能在 `/Fluxon/` 这类项目子路径下正常工作；避免只适用于站点根路径的内部链接
- 在 `README*.md` 中，凡是指向已发布 `.md` 文档页的相对超链接，默认使用 GitHub Pages 绝对 URL，这样从 GitHub 点击时会直接落到已发布站点。例外：顶部 `README.md` 和 `README_CN.md` 之间的语言切换链接保留为仓库相对路径
- 在文档浏览器中，保持左侧树默认展开，包含 `首页`，并把 `roadmap` 放在 `首页` 后面
- 写文档时先给出稳定结论，再展开细节，遵循渐进式展开原则
- 修改 README、用户文档、开发文档或 roadmap 时，默认同步更新中英文版本；设计文档按当前约定可以仅保留中文版，除非任务明确要求英文对应页
- 优先使用自然的工程术语；避免使用“根对象”“第一层分支”“authority object”这类模板化表达
- 对跨语言边界、所有权 / 生命周期规则、异步数据流这类内容，默认补图或表
- 明确区分公共契约、当前实现和专用 fast path
- 仓库级文档规则应保持可复用、技术无关。具体案例教训应放在示例、反例或 review 备注里，而不是规则本身
- 对行为、所有权或性能结论，要明确其作用范围、抽象层、前提条件和排除项
- 不要把局部事实直接泛化成系统级结论；若要上升到整体结论，必须在同一抽象层上补完整条路径
- 文档里默认少用“不是……而是……”。只有当上下文已经建立了两边对比，并且该对比在当前位置确实有助于理解时才使用

## 代码注释
- 代码注释使用英文
- 优先写简短注释，说明函数或代码块在做什么
- 保持注释易于扫读；只有在结构上确实有帮助时才使用 bullets
- 当它确实有助于解释不直观逻辑时，可以使用简洁、结构化的因果链说明，但要避免写成长篇因果式注释

## 公共 API 契约
- 公共 API 必须使用强契约。不要暴露“也许是这个类型，也许是那个类型”的行为
- 面向用户的示例、快速开始、README 和用户文档必须直接调用稳定公共契约
- 不要在面向用户的代码路径中使用 duck-typing、`getattr(...)`、`callable(...)` 或实现探测
- 如果确实需要兼容逻辑，应放在专门的适配层里，而不是放进示例或文档
- 类型签名、文档和运行时行为必须一致。如果 API 说它返回 `MemHolder`，那它就必须返回 `MemHolder`
- 对内部不变量，要快速失败或直接断言。不要像契约不清楚一样静默探测后再回退
- 对一个语义操作，只保留一条主路径。除非这种区别本身就是契约的一部分，否则不要在同一套公开模式里混用 `foo_blocking()` 和 `foo().wait()`
